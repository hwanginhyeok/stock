# Workflow Rules

## 워크플로우 원칙

1. **수집** → **분석** → **생성** → **게시** 순서 엄수
2. 모든 단계의 결과는 DB에 저장
3. 생성된 콘텐츠에는 반드시 면책조항 포함
4. 투자 권유/추천 표현 금지

## 보안 규칙

- API 키 하드코딩 절대 금지 → `.env` 파일로 관리
- `.env` 파일은 `.gitignore`에 포함
- 로그에 API 키, 토큰 등 민감 정보 출력 금지
- 크롤링 시 robots.txt 준수, rate limiting 적용

## TASK 상세 관리

### TASK.md 실시간 갱신 규칙

| 트리거 | 행동 |
|--------|------|
| **작업 착수 시** | "현재 진행 중" 테이블에 등록 + 상태 → `진행` |
| **작업 완료 시** | 작업 현황에서 삭제 → 완료 섹션 최상단에 추가 |
| **새 작업 발견 시** | 즉시 작업 현황에 등록 (대화 중 나온 요청, 버그, 개선 포인트 등) |
| **커밋 직전** | TASK.md 상태가 실제와 일치하는지 점검 |
| **세션 종료 시** | 미완료 작업은 진행 상황 기록, 완료된 건 정리 |

### TASK 상세 로그 작성 기준

`docs/프로젝트/task/{TASK_ID}.md`에 작성하는 경우:

- 설계 결정이 포함된 작업 (왜 A를 선택하고 B를 기각했는지)
- 여러 파일을 변경한 작업 (변경 추적 필요)
- 나중에 "왜 이렇게 했지?" 질문이 예상되는 작업
- 사용자 검토가 필요한 작업 (미결 이슈 기록)
