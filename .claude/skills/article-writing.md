# Article Writing Rules

## 아티클 작성 시 필수 참조 (세션 시작 시 로딩)

아티클 관련 작업을 시작하면 반드시 아래 3개 파일을 먼저 읽는다:

| 순서 | 파일 | 용도 |
|------|------|------|
| 1 | `.claude/rules/writing-voice.md` | 저자의 어조·문체·감정 표현 패턴 |
| 2 | `docs/프로젝트/PUBLISH_SCHEDULE.md` | 게시 스케줄 + 게시 트래커 |
| 3 | `docs/프로젝트/PUBLISH_SCHEDULE.md` `아티클 마스터` | 전체 아티클 현황 + 디렉토리 위치 |

---

## 아티클 생산 파이프라인

| 단계 | 코드 | 설명 | 산출물 |
|------|------|------|--------|
| 1 | 기획 | 주제 선정, 논점 구조 잡기 | 기획 메모 |
| 2 | 리서치 | 자료 수집 + 출처 확보 + 팩트체크 | `research/sources.md` |
| 3 | 초안 | 1차 초안 작성 | `drafts/*_v1.md` |
| 4 | 퇴고 | 문체/논리/팩트 점검, 2차안 작성 | `drafts/*_v2.md` |
| 5 | 시각화 | 차트 생성 + 인터넷 이미지 수집 | `visuals/`, `images/` |
| 6 | 최종 검수 | 마지막 읽기 + X 포맷 변환 | `drafts/*_x_publish.md` |
| 7 | 게시 대기 | published 폴더 이동 | `published/` |

각 단계 완료 시 `PUBLISH_SCHEDULE.md` 아티클 마스터 테이블 상태를 갱신한다.

## 아티클 파일 맵

### 디렉토리 구조 원칙

```

│       └── {순번}_{티커}_{slug}/
│
├── research/                              ← 순수 리서치 (아티클과 분리)
│   ├── stocks/{ticker}/                   ← 종목 펀더멘탈/테제
```

> **단일 소스 원칙**: 아티클 목록·단계·폴더명·게시 현황은 `docs/프로젝트/PUBLISH_SCHEDULE.md`의 **아티클 마스터** 테이블에서 관리한다. 여기에 중복 기록하지 않는다.

## 에디토리얼 디렉터 모델

이 프로젝트는 **사용자 = 에디토리얼 디렉터, AI = 실행팀** 구조로 운영된다.

| 역할 | 사용자 (에디토리얼 디렉터) | AI (실행) |
|------|--------------------------|-----------|
| 테제/관점 | 핵심 테제 제시, 방향 결정 | 테제를 뒷받침하는 리서치 보강 |
| 구조 | 아웃라인 승인/수정 | 아웃라인 초안 제안 |
| 글 작성 | 검토 + 수정 지시 | v1 초안 → v2 퇴고 실행 |
| 시각화 | QA + 승인 | 차트 생성, 이미지 수집 |
| 포맷 변환 | — | x_publish, 네이버 HTML 자동 변환 |
| 게시 | 최종 게시 실행 (수동) | 게시 패키지 준비 |
| 품질 판단 | "게시 가능" 최종 결정 | 6항목 체크리스트 자가 점검 |

**핵심 원칙**: 사용자가 "무엇을 말할지" 결정하고, AI가 "어떻게 만들지" 실행한다.

## 저자의 포지션 & 철학

### 핵심 신념
- 테슬라 및 기술 진보에 대해 **강하게 믿는 편**이다
- 단, 맹목적 믿음이 아니라 **제1원칙으로 검증된 믿음**이다
  - "이 가정이 성립하려면 무엇이 참이어야 하는가?"
  - "어떻게 하면 실현되는가?"
  - "얼마나 걸리는가?"

### 철학적 뿌리: 데이비드 도이치 (David Deutsch)
- "The Beginning of Infinity" — 지식의 성장은 무한하다
- "문제는 반드시 풀린다, 지식이 충분하면"
- 낙관론은 감정이 아니라 **인식론적 입장**이다
- 우리는 무한한 진보의 시작점에 있다
- 인류는 계속 더 나아간다 (Enlightenment, 계몽)

이 철학은 글의 기저 전제다. 명시하지 않더라도 항상 깔려 있다.

## 톤 & 목적

### 독자에게 주고 싶은 것
- **비전의 공유** — "이런 세상이 오고 있다"
- **생각의 확장** — "이렇게 보면 더 멀리 보인다"
- 정보 전달이 아니라 **프레임의 전환**

### 글의 톤
- 확신 있되 강요하지 않는다
- 독자에게 판단을 돌려준다 — "어떻게 보이는가?"
- 냉소나 리스크 중심 서술은 피한다
- "아직 안 된 것"은 **"아직(not yet)"** 으로 다룬다 — 방향은 맞고 시간의 문제

---

## 글쓰기 스타일

### 구조: 논리를 쌓는 스토리텔링
- 결론을 먼저 던지지 않는다
- 독자가 같이 생각하며 결론에 **도달하게** 만든다
- 각 단계가 다음 단계의 근거가 된다

### 제1원칙 적용 패턴
1. 통념을 제시한다
2. 근본 진실로 분해한다
3. Gap을 보여준다
4. Gap에서 가능성을 추론한다

### 문장
- 짧고 명확하게
- 비유와 유추를 적극 활용 (A:B = C:D 구조)
- 데이터는 논리의 뒷받침으로 사용, 나열하지 않는다

## 포맷 방향성

### 기본 방향: 롱폼 아티클
- 스레드처럼 잘게 나누는 방식은 지양한다
- **하나의 글에 깊이 있고 내용이 많은 아티클**을 추구한다
- 논리가 끊기지 않고 흘러야 한다 — 단편이 아닌 완결된 글
- X 아티클(Notes) 기능 활용 — 롱폼으로 발행

### 적정 분량 가이드라인

| 항목 | 기준 | 비고 |
|------|------|------|
| **목표 분량** | **250~300줄** | 이 범위를 넘으면 루즈해진다 — 깊이는 유지하되 군더더기를 잘라낸다 |
| 초과 시 대응 | 시리즈 분리 검토 | 300줄 초과 → "이걸 2편으로 나눌 수 있는가?" 먼저 판단 |

### 플랫폼별 글자 제한

| 플랫폼 | 제한 | 비고 |
|--------|------|------|
| X 포스트 (Premium) | **25,000자** | 우리 아티클 주 발행 채널 (롱폼 포스트) |
| X 포스트 (무료) | 280자 | — |
| 네이버 블로그 | 사실상 무제한 | — |

- 250~300줄 한글 아티클은 약 8,000~12,000자 → X Premium 25,000자 제한 내 여유
- 300줄 초과(~15,000자+) 시 제한에 근접할 수 있으므로 반드시 시리즈 분리 검토
- **실측**: KR-01(304줄) = 9,525자 ✅ / KR-02(471줄) = 18,623자 ⚠️ 제한 근접
- 제한보다 **독자 집중력**이 분량의 실질적 상한선이다

### 좋은 아티클의 기준
- 읽고 나서 "생각이 바뀌었다" 혹은 "보이지 않던 것이 보인다"는 느낌
- 분량이 많더라도 중간에 멈추지 않고 읽히는 글
- 깊이와 흐름이 함께 있는 글

### 게시 플랫폼 전략 (확정 2026-03-01)

**X (주 플랫폼)** → **네이버 블로그 (변환 게시, 선택)**

| 플랫폼 | 역할 | 포맷 | 우선순위 |
|--------|------|------|----------|
| X Notes | 원문 발행 — 투자 커뮤니티 도달, 브랜드 구축 | 롱폼 아티클 + 이미지 | ★ 필수 |
| 네이버 블로그 | SEO 아카이브 — 검색 유입, 2차 도달 | X 원문을 네이버 형식으로 변환 | ☆ 여건 될 때 |

**원칙:**
- 아티클은 X 기준으로 작성한다 — 네이버를 위해 포맷을 바꾸지 않는다
- 네이버는 X에 쌓인 콘텐츠를 아카이빙하는 개념으로 운영
- 무색무취(politics-stat) 홈페이지와는 **완전 별도 관리** — 브랜드·독자층 다름

**게시 워크플로우:**
1. `published/{순번}_{티커}/` 폴더의 md + 이미지로 X Notes 업로드
2. X 소개 트윗 작성 (아티클 링크 포함)
3. (선택) 네이버 블로그에 동일 내용 변환 게시

**네이버 변환 절차:**
1. `python scripts/generate_naver_html.py <*_v2.md>` 실행 — **반드시 v2 (퇴고본) 사용, x_publish 금지**
2. 생성된 HTML에서 blockquote 내 `상태: ...` 메타데이터 라인 제거
3. `_naver.html`을 `naver/` 하위 디렉토리로 이동
4. 이미지를 `naver/`에 번호순 한글 이름으로 복사 (`01_이미지명.png` ...)
5. `naver/README.md` 작성 (이미지 삽입 순서 + 태그 + 체크리스트)
6. 형식·스타일 상세: `.claude/rules/naver-html.md` 참조

> x_publish는 X Notes 전용 (테이블→텍스트 변환 상태). 네이버 변환에 쓰면 HTML 테이블이 생성되지 않는다.

**게시 패키지 디렉토리:**
```
published/{순번}_{티커}_{slug}/
├── *_v2.md                   ← 퇴고본 (네이버 HTML 소스)
├── *_x_publish.md            ← X Notes 게시용
├── X_IMAGE_GUIDE.md          ← X 이미지 삽입 가이드 (📎 번호 → 소스 파일 매핑)
├── sources.md                ← 출처
├── visuals/                  ← 생성 이미지 (matplotlib)
├── images/                   ← 수집 이미지
└── naver/                    ← 네이버 블로그 게시 패키지
    ├── *_naver.html          ← v2에서 변환된 HTML (x_publish 금지)
    ├── README.md             ← 게시 가이드
    ├── 01_이미지명.png        ← 삽입 순서대로 번호 매긴 이미지
    ├── 02_이미지명.png
    └── ...
```

## 시각화 규칙

### 기본 방향
- 텍스트로만 전달하기 어려운 것을 시각화한다
- 글 안에 삽입하는 이미지 — PNG로 생성해 아티클에 첨부
- 스크립트는 아티클 폴더 루트에 `generate_visuals.py`로 저장

### 시각화 유형
| 유형 | 용도 | 도구 |
|------|------|------|
| 타임라인 | 연도별 이벤트 흐름 | matplotlib 수평 플롯 |
| 달성 현황 테이블 | 목표별 상태 색상 코딩 | matplotlib text + patch |
| 카드 | 핵심 개념 비교 | matplotlib subplot |
| 바 차트 | 수치 비교 (gap, 성장률) | matplotlib barh |

### 디자인 기준
- 배경: `#0d1117` (어두운 GitHub 스타일)
- 카드 배경: `#161b22`
- 색상 코드 — 상태별:
  - 완료/성장: `#3fb950` (green)
  - 부분/지연: `#d29922` (yellow)
  - 진행중: `#58a6ff` (blue)
  - 미완료: `#f85149` (red)
  - 전략변경: `#db6d28` (orange)
  - 미착수: `#6e7681` (gray)
- 해상도: `dpi=180`
- 한글 폰트: `NotoSansCJK-Regular.ttc` → family `"Noto Sans CJK JP"`

### 폰트 설정 (필수 패턴)
```python
import matplotlib.font_manager as fm
_FONT_PATH = "/usr/share/fonts/opentype/noto/NotoSansCJK-Regular.ttc"
if os.path.exists(_FONT_PATH):
    fm.fontManager.addfont(_FONT_PATH)
    plt.rcParams["font.family"] = "Noto Sans CJK JP"
plt.rcParams["axes.unicode_minus"] = False
```

### 저장 위치
```
data/articles/{순번}_{티커}_{slug}/
├── generate_visuals.py   ← 생성 스크립트
└── visuals/
    ├── 01_*.png
    ├── 02_*.png
    └── 03_*.png
```

### 이미지 생성 후 QA 체크리스트 (필수)

이미지 생성 후 반드시 아래 항목을 코드로 검토하고 문제가 있으면 수정 후 재생성한다.

| 항목 | 확인 방법 | 기준 |
|------|-----------|------|
| 텍스트 잘림 | bbox 좌표가 figure 범위(0~1) 밖인지 | x < 0, x > 1, y < 0, y > 1 이면 수정 |
| 텍스트 겹침 | 인접 요소 bbox 간 overlap 여부 | overlap > 5px 이면 위치 조정 |
| 여백 | 최상단/하단 텍스트와 figure 경계 간격 | 최소 2% 여백 확보 |
| 행 넘침 | 테이블 마지막 행이 하단 경계 안에 있는지 | body_bot(0.04) 위에 위치해야 함 |
| 파일 크기 | PNG 파일이 정상 생성됐는지 | > 10KB |

**QA는 `generate_visuals.py` 안의 `qa_check()` 함수로 자동화한다.**
생성 스크립트 마지막에 `qa_check()` 호출 필수.

## 출처 & 팩트체크 규칙

### 원칙
- **모든 수치, 인용문, 팩트에는 출처가 있어야 한다**
- 출처 없는 주장은 글에 넣지 않는다
- 리서치 단계에서 출처를 수집하고, 초안 작성 전에 검증한다

### 리서치 문서 (필수)

아티클 작업 시 `research/sources.md`를 반드시 생성한다.

```
data/articles/{순번}_{티커}_{slug}/
├── research/
```

### sources.md 구조

```markdown
# 출처 & 팩트체크

## 1차 자료 (공식 문서/발표)
| # | 자료명 | URL | 인용 위치 |

## 데이터 & 수치
| # | 주장 | 출처 | 검증 |

## 인용문
| # | 인용 | 출처 | 검증 |
```

### 아티클 내 출처 표기

- 글 하단에 `## 참고 자료` 섹션 추가
- 1차 자료(공식 문서)와 데이터 출처를 구분하여 나열
- X Notes 게시용에도 동일하게 포함

### 팩트체크 기준

| 유형 | 기준 | 예시 |
|------|------|------|
| 수치 | 공식 보고서 또는 신뢰 매체 2곳 이상 교차 확인 | 매출, 판매량, 시장 점유율 |
| 인용문 | 원문 확인 (영상/문서 직접 확인) | 머스크 발언, 도이치 저서 |
| 사건/날짜 | 공식 발표 또는 보도 확인 | 출시일, 서비스 개시일 |
| 전망/추정 | 출처 + "추정" 명시 | 애널리스트 전망, 시장 예측 |

### X Notes 포맷 규칙

- 마크다운 테이블(`| | |`)은 X Notes에서 렌더링되지 않음
- 원본(v2)과 게시용(x_publish)을 분리
- 게시용에서는 테이블 → 볼드 라벨 + 이모지 리스트로 변환

```
## 면책조항

이 글은 특정 종목의 매수·매도를 권유하지 않는다. 모든 투자 판단과 책임은 본인에게 있다.
```

| 항목 | 규칙 |
|------|------|
| 순서 | **참고 자료 → 면책조항** (면책이 맨 마지막) |
| 면책조항 | `## 면책조항` 섹션 헤더 + 1줄 본문. 블록쿼트/이탤릭 금지 |
| 다음 편 예고 | 시리즈물에만. 참고 자료 이전에 배치. **아래 작성 원칙 필수 준수** |
| 구분선 | 각 섹션 사이에 `---` 하나씩 |

### 다음 편 예고 작성 원칙

1. **실제 다음 편과 일치해야 한다** — 예고한 내용이 다음 편에서 실제로 다뤄져야 한다. 예고와 다음 편이 어긋나면 독자 신뢰가 무너진다.
2. **현재 글의 핵심 프레임을 확장한다** — 다음 편의 주제를 단순히 던지지 않는다. 현재 글의 논리가 자연스럽게 다음 글로 이어지는 **논리적 다리(bridge)**를 만든다.
3. **현재 글의 결론에서 출발한다** — 본문 마지막 단락의 핵심 개념을 잡고, 거기서 다음 질문을 끌어낸다.
4. **마지막 한 문장은 질문으로 끝낸다** — 독자가 다음 편을 읽고 싶게 만드는 훅(hook).

**예시 (KR-01 → KR-02):**
- KR-01의 프레임: "병목"
- 확장: 반도체 병목 → 전력 병목 → **데이터 병목** → Physical AI가 와야 데이터가 나온다 → 현대차가 $110B을 걸었다
- 훅: "하드웨어를 만드는 것과 가치를 가져가는 것은 같은 일인가."
