# 아티클 퇴고 (article-revision)

## 목적

- 초안(v1)을 완성도 높은 v2로 다듬고, X Notes 게시용(x_publish)까지 변환한다
- 퇴고 기준 체크리스트를 통해 일관된 품질을 보장한다
- 사용자 피드백을 체계적으로 추적하고, 반복 패턴을 원칙으로 승격한다

## 실행 조건

| 트리거 | 설명 |
|--------|------|
| 초안(v1) 완성 후 | 자가 퇴고 시작 |
| 사용자 퇴고 피드백 수령 | 피드백 반영 퇴고 |
| 게시 직전 최종 점검 | 최종 퇴고 (x_publish 생성 포함) |

## 절차

### 0단계: 사전 로딩 (필수)

아래 3개 파일을 **반드시** 먼저 읽는다. 순서대로.

| 순서 | 파일 | 용도 |
|------|------|------|
| 1 | `.claude/rules/writing-voice.md` | 저자의 어조·문체·감정 패턴 + 퇴고 피드백 로그 |
| 2 | `.claude/rules/article-writing.md` | 아티클 구조·포맷·하단 구조·시각화 규칙 |
| 3 | `docs/프로젝트/task/4-x.md` | 해당 아티클의 퇴고 이력 (없으면 생성) |

### 1단계: 현재 상태 판단 (의사결정 트리)

```
어떤 퇴고인가?
│
├─ A. 자가 퇴고 (v1 → v2)
│     초안 전체를 체크리스트 6항목으로 검토
│     → v2 작성 → 2단계
│
├─ B. 피드백 반영 (사용자 피드백 → v2 수정)
│     피드백 항목별로 변경 적용
│     → 퇴고 이력 기록 → 2단계
│
└─ C. 최종 퇴고 (v2 → x_publish)
      마지막 읽기 + X Notes 포맷 변환
      → x_publish 생성 → 3단계
```

### 2단계: 퇴고 기준 체크리스트 (6항목)

v1→v2, 또는 피드백 반영 시 아래 6가지를 점검한다:

| # | 항목 | 점검 내용 | 참조 |
|---|------|-----------|------|
| 1 | **어조** | "~다"체, 직접 감탄 없음, 수치로 감정 전달, 독자에게 묻는 톤 | writing-voice.md |
| 2 | **논리 흐름** | 통념→분해→Gap→가능성 패턴, 결론 먼저 던지지 않음, 각 단계가 다음의 근거 | writing-voice.md |
| 2.5 | **시각적 호흡** | 흐름 전환·인용 발언("") 앞뒤에 빈 줄 2~3개, 남용 금지 | writing-voice.md |
| 3 | **팩트 & 출처** | 모든 수치에 출처 있음, sources.md와 대조, 교차 검증 2곳 이상 | article-writing.md |
| 4 | **데이터 배치** | 데이터 나열 아님, 논리 흐름 속 정확한 위치에 배치, 비유 활용 | writing-voice.md |
| 5 | **하단 구조** | 순서: 본문끝 → (예고편) → --- → 참고자료 → --- → 면책조항 | article-writing.md |
| 6 | **예고편** | 시리즈물만. 현재 글 프레임 확장, 논리적 다리, 마지막 문장은 질문 | article-writing.md |

### 3단계: v2 → x_publish 변환

X Notes에서 렌더링되지 않는 요소를 변환한다:

#### 테이블 → 볼드+이모지 리스트

```markdown
# 변환 전 (v2, 마크다운 테이블)
| 항목 | 수치 | 비고 |
|------|------|------|
| 매출 | $100B | YoY +30% |
| 이익 | $20B | 마진 20% |

# 변환 후 (x_publish)
**매출** — $100B (YoY +30%)
**이익** — $20B (마진 20%)
```

#### 기타 변환 규칙

| 요소 | v2 (원본) | x_publish (게시용) |
|------|-----------|-------------------|
| 테이블 | `\| ... \|` 마크다운 | 볼드 라벨 + 이모지 리스트 |
| 코드 블록 | ` ``` ` | 제거 또는 인라인 설명 |
| 각주 | `[^1]` 스타일 | 본문 괄호 삽입 또는 하단 나열 |
| 이미지 마커 | 그대로 유지 | `📎 *[이미지: 설명]*` 유지 |
| 시각적 호흡 (빈 줄) | 그대로 유지 | **반드시 유지** — 흐름 전환·인용 발언("") 앞뒤의 연속 빈 줄 보존 |

#### X 이미지 삽입 가이드 생성 (필수)

x_publish 생성 시 `X_IMAGE_GUIDE.md`를 함께 만든다. 📎 마커 번호 → 실제 소스 파일 매핑표.

```markdown
# X Notes 이미지 삽입 가이드 — {아티클 제목}

## 이미지 삽입 순서

| 📎 번호 | 소스 파일 | 본문 위치 |
|---------|-----------|-----------|
| 1 | `visuals/01_*.png` 또는 `images/*` | §N 전/후 |
| 2 | ... | ... |

## 게시 절차
- [ ] X Notes 새 아티클 생성
- [ ] x_publish.md 내용 붙여넣기
- [ ] 📎 마커 위치에 위 순서대로 이미지 삽입
- [ ] 면책조항 확인
- [ ] 소개 트윗 작성
- [ ] 모바일 렌더링 확인
- [ ] 발행
```

> 📎 번호와 파일명 번호는 다를 수 있다 (삽입 순서 ≠ 생성 순서). 반드시 매핑표를 만든다.

#### 저장 위치

```
data/articles/{순번}_{티커}_{slug}/
└── drafts/
    ├── *_v1.md            ← 초안 (보존)
    ├── *_v2.md            ← 퇴고본 (마크다운 테이블 포함)
    └── *_x_publish.md     ← X Notes 게시용 (테이블 변환 완료)
```

published 이동 시:
```
published/{순번}_{티커}_{slug}/
├── *_v2.md
├── *_x_publish.md
├── X_IMAGE_GUIDE.md       ← 📎 번호 → 소스 파일 매핑 + 게시 체크리스트
├── visuals/
├── images/
└── naver/
```

### 4단계: 퇴고 이력 기록

`docs/프로젝트/task/4-x.md`의 `## 퇴고 이력` 섹션에 기록:

```markdown
### {날짜} — {퇴고 유형}

**피드백/점검 내용:**
- {항목 1}
- {항목 2}

**변경 결정:**
- {어디를, 왜, 어떻게 바꿨는가}

**보류 항목:**
- {다음 버전으로 넘긴 것과 이유}
```

### 5단계: 피드백 패턴 추적

1. 현재 피드백이 `writing-voice.md`의 `퇴고 피드백 로그`에 이미 있는 유형인지 확인
2. 같은 유형이면 카운트 누적
3. **3회 이상 반복** → `반복 패턴 메모`에 기록
4. 패턴 확인되면 → `article-writing.md` 또는 `writing-voice.md`에 원칙으로 승격 제안

### 6단계: 상태 갱신 → schedule-sync

퇴고 완료 시:
- v2 완성 → `schedule-sync` 실행 (상태: "퇴고 완료" / "게시 가능")
- x_publish 완성 → `schedule-sync` 실행 (상태: "게시 가능")
- → `generate-visuals` (시각화 미제작 시)
- → `publish-x` (게시 준비 완료 시)

## 퇴고 유형별 체크리스트 요약

| 유형 | 체크 항목 | 산출물 | 다음 스킬 |
|------|-----------|--------|-----------|
| A. 자가 퇴고 | 6항목 전부 | v2 | schedule-sync |
| B. 피드백 반영 | 피드백 항목 + 재점검 | v2 수정 | schedule-sync |
| C. 최종 퇴고 | 최종 읽기 + 변환 | x_publish | generate-visuals → publish-x |

## 주의사항

- **v1은 절대 덮어쓰지 않는다** — v2를 별도 파일로 생성
- writing-voice.md의 "하지 않는 것" 목록을 매번 확인한다 (냉소, 경어체, 직접 감탄 등)
- 테이블 → 텍스트 변환 시 **정보 손실이 없는지** 확인 (컬럼 수가 많은 테이블은 핵심만 선별)
- 면책조항은 **항상 맨 마지막** — 참고 자료 아래
- 예고편은 단순 주제 예고가 아니라 **현재 글 프레임의 논리적 확장**이어야 한다
- task 파일(4-x.md)이 없으면 먼저 생성한다 (workflow.md의 아티클 태스크 필수 규칙)
