# 아티클 초안 작성 (article-drafting)

## 목적

- 리서치 완료(Phase 4) → 아웃라인 → v1 초안까지의 흐름을 정의한다
- "리서치는 됐는데 글이 안 써지는" 병목을 해소한다
- 아웃라인 없이 v1을 쓰지 않는다 — 구조가 먼저, 문장은 나중

## 실행 조건

| 트리거 | 설명 |
|--------|------|
| article-research Phase 4 완료 후 | 리서치·시각화 계획까지 끝난 상태 |
| 사용자가 "초안 쓰자" 지시 | 리서치가 충분하다고 판단한 시점 |

## 절차

### 0단계: 사전 로딩 (필수)

| 순서 | 파일 | 용도 |
|------|------|------|
| 1 | `.claude/rules/writing-voice.md` | 어조·문체·논리 전개 패턴 |
| 2 | `.claude/rules/article-writing.md` | 아티클 구조·포맷·분량·하단 구조 |
| 3 | `research/thesis.md` | 핵심 테제 확인 |
| 4 | `research/claims.md` | 검증된 주장 목록 |
| 5 | `research/sources.md` | 출처 목록 |

### 1단계: 아웃라인 작성 — 담당: **AI 초안 → 사용자 승인**

리서치 결과를 **글의 구조**로 변환한다. 아웃라인 = 논리의 뼈대.

#### 1.1 핵심 질문 정의

아티클이 답해야 하는 **하나의 질문**을 정한다.

```
이 글을 읽은 독자가 갖게 될 질문: "_______________?"
이 글이 주는 답: "_______________."
```

- 질문이 2개 이상이면 시리즈 분리를 검토한다
- 질문은 구체적이어야 한다 — "테슬라는 좋은가?" ❌ → "테슬라의 수직통합이 마진에 미치는 영향은?" ✅

#### 1.2 논리 흐름 설계

writing-voice.md의 **통념 → 분해 → Gap → 가능성** 패턴을 적용한다:

```
1. 도입: 통념 또는 현재 상황 제시 (독자가 아는 것에서 시작)
2. 분해: 통념을 제1원칙으로 해체 (왜 그런가? 정말 그런가?)
3. Gap: 통념과 현실의 괴리 (데이터로 보여주기)
4. 전개: Gap에서 가능성 추론 (여기가 글의 핵심)
5. 결론: 독자에게 질문 돌려주기 (판단은 당신의 몫)
```

모든 아티클이 이 5단계를 따를 필요는 없다. 주제에 맞게 변형한다.
단, **결론을 먼저 던지지 않는다** — 이것만은 불변.

#### 1.3 섹션 구조화

```markdown
# {제목}

> 상태: 아웃라인 / 작성자 / 날짜

## 핵심 질문
- {1.1에서 정한 질문}

## 섹션 구조

### §1 {도입 — 통념/현황}
- 핵심 포인트: ...
- 사용할 데이터: ... (claims.md #{번호})
- 예상 분량: ~40줄

### §2 {분해 — 제1원칙}
- 핵심 포인트: ...
- 사용할 데이터: ...
- 비유/유추: ...
- 예상 분량: ~50줄

### §3 {Gap — 데이터로 보여주기}
- 핵심 포인트: ...
- 시각화: 📎 {visuals/0x_*.png} (있으면)
- 예상 분량: ~60줄

### §4 {전개 — 가능성 추론}
- 핵심 포인트: ...
- 사용할 데이터: ...
- 예상 분량: ~60줄

### §5 {결론 — 질문 돌려주기}
- 마무리 방향: ...
- (시리즈물) 다음 편 브릿지: ...
- 예상 분량: ~30줄

## 총 예상 분량: ~240줄

## 📎 시각화 배치 계획
| 위치 | 파일 | 설명 |
|------|------|------|
| §3 뒤 | visuals/01_*.png | ... |
| §4 중간 | visuals/02_*.png | ... |
```

#### 1.4 분량 점검

| 기준 | 값 | 초과 시 |
|------|-----|---------|
| 목표 | 250~300줄 | — |
| 상한 | 300줄 | 시리즈 분리 검토 |
| 글자수 상한 | ~12,000자 | X Premium 25,000자 내 여유 확인 |

섹션별 예상 분량 합이 300줄을 넘으면 **아웃라인 단계에서** 분리한다.
v1 쓰고 나서 분리하면 시간 낭비.

#### 1.5 아웃라인 저장

```
data/articles/{순번}_{티커}_{slug}/drafts/outline.md
```

### 2단계: 사용자 검토

아웃라인을 사용자에게 보여주고 승인을 받는다.

검토 포인트:
- 핵심 질문이 맞는가?
- 논리 흐름이 자연스러운가?
- 빠진 포인트가 있는가?
- 분량 배분이 적절한가?

**승인 없이 v1 작성에 들어가지 않는다.**

### 3단계: v1 초안 작성 — 담당: **AI** (사용자 검토)

아웃라인을 문장으로 풀어쓴다.

#### 3.1 작성 원칙

| 원칙 | 설명 |
|------|------|
| 아웃라인 순서를 따른다 | 쓰다가 구조를 바꾸지 않는다. 구조를 바꾸려면 아웃라인을 먼저 수정 |
| 데이터는 claims.md 번호로 참조 | 출처 추적이 가능하도록 |
| 📎 마커를 배치한다 | 시각화 위치에 `📎 *[이미지: 설명]*` 삽입 → `image-markers` 스킬 참조 |
| 시각적 호흡을 넣는다 | 논점 전환, 인용 발언("") 앞뒤에 빈 줄 2~3개 |
| 하단 구조를 포함한다 | 본문끝 → (예고편) → --- → 참고자료 → --- → 면책조항 |

#### 3.2 섹션별 작성 흐름

각 섹션을 쓸 때:
1. 아웃라인의 "핵심 포인트"를 첫 문장으로 시작
2. 뒷받침 데이터를 논리 흐름 속에 배치 (나열 금지)
3. 섹션 끝에서 다음 섹션으로의 논리적 다리 확인
4. 예상 분량 ±20% 이내 유지

#### 3.3 저장

```
data/articles/{순번}_{티커}_{slug}/drafts/{slug}_v1.md
```

### 4단계: v1 자가 점검

v1 작성 후 즉시 체크 (퇴고가 아닌 기본 점검):

| # | 항목 | 확인 |
|---|------|------|
| 1 | 아웃라인 구조와 일치하는가 | 섹션 순서, 핵심 포인트 누락 여부 |
| 2 | 분량이 250~300줄 범위인가 | `wc -l` 확인 |
| 3 | 📎 마커가 시각화 계획과 일치하는가 | 개수, 위치 |
| 4 | 하단 구조가 올바른가 | 참고자료 → 면책조항 순서 |
| 5 | 면책조항이 포함되었는가 | — |

통과 시 → `article-revision` 스킬로 넘긴다.

## 다음 스킬 연결

```
article-research (Phase 4) → [이 스킬] → article-revision (퇴고)
                                  ↓
                           image-markers (📎 배치)
```

## 주의사항

- **아웃라인 없이 v1을 쓰지 않는다** — 가장 중요한 원칙
- 아웃라인 단계에서 분량 초과를 잡아야 한다 — v1 쓰고 나서 자르면 시간 낭비
- claims.md에 없는 수치는 v1에 넣지 않는다 — 출처 없는 주장 금지
- v1은 "완벽한 글"이 아니라 "구조가 잡힌 초안"이다 — 문체는 퇴고에서 다듬는다
- 시리즈물의 다음 편 예고는 v1에서 초안을 넣되, 퇴고에서 정제한다
