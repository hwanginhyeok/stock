# 아티클 게시 리뷰 (publish-review)

## 목적

- 아티클을 게시하기 전에 **체계적으로 리뷰**한다
- 8개 차원별로 0~10 점수를 매기고, 10이 아닌 항목은 **왜 10이 아닌지** 설명한 뒤 수정 제안
- /plan-ceo-review, /plan-eng-review와 같은 인터랙티브 구조

## 실행 조건

| 트리거 | 설명 |
|--------|------|
| "리뷰해줘", "게시 리뷰" | 사용자가 아티클 리뷰 요청 |
| v1 완료 후 | 자가 퇴고 전 전체 체크 |
| v2 완료 후 | 게시 전 최종 체크 |
| "publish-review" | 직접 호출 |

## 사전 로딩 (필수)

리뷰 시작 전 반드시 읽는다:

| 순서 | 파일 | 용도 |
|------|------|------|
| 1 | `.claude/skills/writing-voice.md` | 저자 DNA — 어조, 문체, 감정, 논리 패턴 |
| 2 | `.claude/skills/article-writing.md` | 구조, 포맷, 팩트체크, 하단 구조 규칙 |
| 3 | `.claude/skills/article-shipping.md` | WIP 제한, 타임박스 확인 |
| 4 | 해당 아티클의 `research/sources.md` | 출처 대조용 |

## 리뷰 프로세스

### 1단계: 전체 읽기 + 스코어카드

아티클 전문을 읽고 8개 차원별 점수를 매긴다.

```
┌─────────────────────────────────────────┐
│         PUBLISH REVIEW SCORECARD        │
├──────────────────┬──────┬───────────────┤
│ 차원             │ 점수 │ 판정          │
├──────────────────┼──────┼───────────────┤
│ 1. 테제          │ ?/10 │               │
│ 2. 논리 흐름     │ ?/10 │               │
│ 3. 어조 & 문체   │ ?/10 │               │
│ 4. 호흡 & 리듬   │ ?/10 │               │
│ 5. 데이터 배치   │ ?/10 │               │
│ 6. 팩트 & 출처   │ ?/10 │               │
│ 7. 구조 & 포맷   │ ?/10 │               │
│ 8. 예고편 & 연결 │ ?/10 │               │
├──────────────────┼──────┼───────────────┤
│ 종합             │ ?/10 │               │
└──────────────────┴──────┴───────────────┘
```

### 2단계: 차원별 상세 리뷰 (인터랙티브)

**10이 아닌 차원만** 하나씩 사용자에게 보여주고 논의한다.

각 차원에 대해:
1. 현재 점수와 이유 설명
2. **10이 되려면 무엇이 필요한지** 구체적으로 제시
3. 수정 제안 (구체적 문장/구간 지시)
4. 사용자 판단: 수정 / 유지 / 보류

**8점 이상인 차원은 간략히만**, 7점 이하인 차원은 상세히 다룬다.

### 3단계: 수정 실행 + 피드백 시스템 연동

사용자가 수정을 승인하면:
1. 아티클에 수정 적용
2. **feedback-tracking 5단계** 실행 — 수정 이유를 묻고, 규칙 파일 업데이트
3. 수정 후 해당 차원 재채점

### 4단계: 최종 판정

```
VERDICT: SHIP / REVISE / HOLD

SHIP   — 종합 8/10 이상, 7점 이하 차원 0개
REVISE — 종합 6~7/10, 또는 7점 이하 차원 1~2개
HOLD   — 종합 5/10 이하, 또는 팩트/출처에 심각한 문제
```

## 8개 리뷰 차원 상세

### 1. 테제 (Thesis)

| 점검 | 기준 |
|------|------|
| 핵심 테제가 1문장으로 추출 가능한가 | "이 글의 테제는 ___이다" |
| 테제가 글 전체를 관통하는가 | 모든 섹션이 테제를 뒷받침하거나 확장 |
| 테제가 독창적인가 | 남들이 이미 다 하는 말이 아닌가 |
| 비유/유추가 테제를 강화하는가 | A:B = C:D 구조가 있으면 가산점 |

### 2. 논리 흐름 (Logic Flow)

| 점검 | 기준 |
|------|------|
| 통념→분해→Gap→가능성 패턴 | 또는 변형이 자연스러운가 |
| 결론을 먼저 던지지 않았는가 | 독자가 도달하게 만드는가 |
| 각 섹션이 다음 섹션의 근거가 되는가 | 논리적 다리가 있는가 |
| 비약이 없는가 | 데이터 없이 주장하는 구간 |

### 3. 어조 & 문체 (Voice & Style)

| 점검 | 기준 |
|------|------|
| "~다" 체 일관성 | 경어체 혼입 0건 |
| 직접 감탄 없음 | "대단하다", "놀랍다" 0건 |
| 수치로 감정 전달 | 숫자가 말하게 하는가 |
| 독자에게 묻는 톤 | 질문형이 적절히 배치됐는가 |
| writing-voice.md "하지 않는 것" 위반 | 0건이어야 함 |

### 4. 호흡 & 리듬 (Breath & Rhythm)

| 점검 | 기준 |
|------|------|
| 내러티브 구간 호흡 | 이야기 구간에서 문장이 충분히 이어지는가 |
| 종결어미 단조로움 | "~했다. ~였다. ~갔다." 3회 이상 연속 없음 |
| 강조 도구 다양성 | 8가지 도구 중 3가지 이상 사용 |
| 도치/강조 구문 과다 | 같은 기법 2회 이상 연속 없음 |
| 시각적 호흡 (빈 줄) | 인용·전환 앞뒤 적절, 남용 없음 |
| 분석 vs 내러티브 구간별 호흡 차이 | 구간에 따라 달라지는가 |

### 5. 데이터 배치 (Data Placement)

| 점검 | 기준 |
|------|------|
| 데이터 나열 없음 | 표만 쏟아붓기가 아닌가 |
| 논리 흐름 속 배치 | 데이터가 필요한 위치에 있는가 |
| 비유로 스케일 전달 | 큰 숫자를 독자가 체감하게 만드는가 |
| 테이블 적정 개수 | 250~300줄 기준 2~4개 |

### 6. 팩트 & 출처 (Facts & Sources)

| 점검 | 기준 |
|------|------|
| 모든 수치에 출처 있음 | sources.md와 대조 |
| 교차 검증 2곳 이상 | 핵심 수치는 반드시 |
| 최신성 | 게시 시점 기준 최신 데이터인가 |
| 추정치 명시 | "추정", "전망" 표기 |
| 인용문 원문 확인 | 영상/문서 직접 확인 |

### 7. 구조 & 포맷 (Structure & Format)

| 점검 | 기준 |
|------|------|
| 분량 250~300줄 | 초과 시 시리즈 분리 검토 |
| 하단 구조 순서 | 본문끝 → 예고편 → --- → 참고자료 → --- → 면책조항 |
| 📎 마커 정합성 | 개수, 위치, 매핑 |
| 면책조항 포함 | 맨 마지막 |
| X 글자수 25,000자 이내 | 실측 확인 |

### 8. 예고편 & 연결 (Preview & Bridge)

| 점검 | 기준 |
|------|------|
| 현재 글 프레임 확장 | 단순 주제 예고가 아닌 논리적 다리 |
| 실제 다음 편과 일치 | 예고 내용이 다음 편에서 다뤄지는가 |
| 마지막 문장이 질문 | 독자가 다음 편을 읽고 싶게 |
| 시리즈 아크 일관성 | 전체 시리즈에서 이 편의 위치가 명확한가 |

## 리뷰 결과 기록

리뷰 완료 후 `docs/프로젝트/task/4-x.md`에 기록:

```markdown
### {날짜} — Publish Review

**스코어카드:**
| 차원 | 점수 | 주요 이슈 |
|------|------|----------|
| 테제 | ?/10 | ... |
| ... | ... | ... |
| **종합** | **?/10** | |

**판정:** SHIP / REVISE / HOLD
**수정 사항:** {N}건 수정, {M}건 보류
**규칙 업데이트:** {업데이트된 파일 목록}
```

## 연동

```
article-drafting (v1 완료)
    → [이 스킬] 1차 리뷰
    → article-revision (퇴고)
    → [이 스킬] 2차 리뷰 (게시 전 최종)
    → feedback-tracking (수정 → 이유 질문 → 규칙 업데이트)
    → article-shipping (SHIP 판정 → 게시)
```

## 주의사항

- **SHIP 판정이 아니면 게시하지 않는다** — 단, 사용자가 명시적으로 오버라이드 가능
- 리뷰어는 **독자 관점**에서 읽는다 — 작성자가 아니라 처음 읽는 사람의 눈으로
- 점수에 관대하지 않는다 — 8점은 "좋다"가 아니라 "10이 아닌 이유가 2개 있다"
- 수정 제안은 **구체적 문장/줄 번호**로 — "톤이 좀 그래요"는 피드백이 아니다
