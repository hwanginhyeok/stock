# 퇴고 피드백 추적 (feedback-tracking)

## 목적

- 사용자의 퇴고 피드백을 체계적으로 기록·분류·추적한다
- 반복 패턴을 감지하고, 3회 이상 반복되면 원칙으로 승격한다
- 피드백이 "기억"에서 사라지지 않게 한다

## 실행 조건

| 트리거 | 설명 |
|--------|------|
| 사용자가 아티클 피드백을 줄 때 | 퇴고 과정에서 수정 요청 수령 시 |
| 퇴고 완료 후 | article-revision 4~5단계에서 호출 |
| 정기 점검 | 세션 시작 시 반복 패턴 메모 확인 |

## 절차

### 1단계: 피드백 기록

피드백을 수령하면 **즉시** `docs/프로젝트/task/4-x.md`에 기록한다.

#### 기록 템플릿

```markdown
### {날짜} — {퇴고 유형: 자가/피드백/최종}

**피드백 원문:**
> {사용자 피드백 그대로 인용}

**카테고리:** {아래 분류표에서 선택}

**변경 결정:**
- {어디를}: {섹션/줄 번호}
- {왜}: {피드백의 핵심 이유}
- {어떻게}: {구체적 변경 내용}

**보류 항목:**
- {다음 버전으로 넘긴 것과 이유}
```

### 2단계: 피드백 분류

모든 피드백을 아래 카테고리 중 하나로 분류한다:

| 코드 | 카테고리 | 예시 |
|------|---------|------|
| `T` | **톤/어조** | "너무 딱딱하다", "경어체 섞였다", "감탄 표현 빼라" |
| `L` | **논리/구조** | "흐름이 끊긴다", "결론을 먼저 던졌다", "섹션 순서 바꿔라" |
| `D` | **데이터/팩트** | "수치 틀렸다", "출처 없다", "데이터 나열이다" |
| `V` | **시각적 표현** | "빈 줄 넣어라", "비유가 안 맞다", "문장이 길다" |
| `F` | **포맷/구조** | "하단 순서 틀렸다", "면책조항 빠졌다", "예고편 불일치" |
| `S` | **범위/분량** | "너무 길다", "이건 다음 편으로", "여기가 핵심이니 더 써라" |

### 3단계: writing-voice.md 퇴고 피드백 로그 갱신

`writing-voice.md` 하단의 **퇴고 피드백 로그** 테이블에 한 줄 추가:

```markdown
| {날짜} | {아티클명} | {피드백 요약 1줄} | {반영 여부: ✅/⏳/❌} |
```

### 4단계: 반복 패턴 감지

피드백 로그를 스캔하여 같은 **카테고리 + 유사 내용**이 있는지 확인한다.

#### 카운트 방법

```
같은 카테고리 코드(T/L/D/V/F/S) + 핵심 키워드 일치 → 동일 유형으로 카운트
```

예시:
- 1회: "결론을 먼저 던졌다" (L) — KR-01
- 2회: "도입부에서 답을 줬다" (L) — PLTR-1편
- 3회: "첫 문단에 결론이 있다" (L) — HIMS → **패턴 확정!**

#### 감지 결과 기록

`writing-voice.md` 하단의 **반복 패턴 메모**에 기록:

```markdown
| 카테고리 | 패턴 | 누적 횟수 | 해당 아티클 | 승격 여부 |
|---------|------|-----------|-----------|----------|
| L | 도입부에서 결론 선제시 | 3 | KR-01, PLTR-1, HIMS | ✅ → writing-voice.md |
```

### 5단계: 즉시 반영 (Instant Update)

**사용자가 수정을 지시하면**, 그 수정이 단순 오타가 아닌 한 **즉시 품질 시스템에 반영**한다.

#### 트리거: 사용자가 아티클을 직접 수정했을 때

사용자가 v1/v2의 문장, 구조, 표현을 직접 고치면 — 그 수정 자체가 피드백이다.

#### 반영 절차

1. 사용자가 아티클을 직접 수정 (또는 수정 지시)
2. AI가 수정 내용을 감지
3. **AI가 묻는다**: "이 부분을 바꾸셨는데, 기존 표현에서 뭐가 별로였어요?" (1문장 질문)
4. 사용자 답변을 받으면:
   - 아티클에 수정 적용 (이미 안 되었으면)
   - **해당 규칙 파일 즉시 업데이트** (writing-voice.md 또는 article-writing.md)
   - task/4-x.md 퇴고 이력에 기록
   - "수정 반영 + {파일명} 규칙 업데이트 완료" 보고
5. 사용자가 "그냥 수정이야" / "별 이유 없어"라고 하면 → 기록만 하고 규칙 업데이트 안 함

#### 질문 예시

```
사용자가 "SpaceX가 문을 열면" → "SpaceX가 길을 닦으면"으로 수정
→ AI: "비유를 바꾸셨는데, '문을 열면'이 뭐가 별로였어요?"

사용자가 §4의 문장 순서를 바꿈
→ AI: "§4 순서를 바꾸셨는데, 기존 흐름에서 뭐가 걸렸어요?"
```

**핵심**: 수정 행위 → AI가 이유를 물어봄 → 이유가 있으면 시스템에 반영. 같은 실수를 두 번 하지 않는다.

### 6단계: 패턴 승격 (3회 규칙)

기존의 **3회 이상 반복** 패턴 승격은 **5단계 즉시 반영과 별도로** 유지한다.

#### 승격 대상 파일

| 카테고리 | 승격 위치 |
|---------|----------|
| `T` (톤/어조) | `writing-voice.md` → 해당 섹션 |
| `L` (논리/구조) | `writing-voice.md` → 논리 전개 섹션 |
| `D` (데이터/팩트) | `article-writing.md` → 출처 & 팩트체크 규칙 |
| `V` (시각적 표현) | `writing-voice.md` → 시각적 호흡 섹션 |
| `F` (포맷/구조) | `article-writing.md` → 해당 섹션 |
| `S` (범위/분량) | `article-writing.md` → 분량 가이드라인 |

#### 승격 절차

1. 패턴을 1~2문장의 **규칙**으로 정제
2. 해당 파일의 적절한 섹션에 추가
3. 반복 패턴 메모의 "승격 여부"를 `✅ → {파일명}` 으로 갱신
4. 사용자에게 "이 패턴을 원칙으로 추가했습니다" 보고

## 세션 시작 시 점검 루틴

매 세션 시작 시 아래를 빠르게 확인:

1. `writing-voice.md` → 반복 패턴 메모에 미승격 항목이 있는가?
2. 2회 누적된 패턴이 있는가? → 다음 퇴고 시 주의 깊게 관찰
3. 최근 승격된 원칙이 있는가? → 이번 아티클에 적용

## 연동

```
article-revision (퇴고)
    ├── 4단계: 퇴고 이력 기록 → [이 스킬] 1~3단계
    └── 5단계: 피드백 패턴 추적 → [이 스킬] 4~5단계
```

## 주의사항

- **피드백은 수령 즉시 기록한다** — "나중에 정리하자"는 누락의 시작
- 사용자 피드백 원문을 반드시 보존한다 — 요약만 남기면 맥락이 사라진다
- 승격은 3회 이상에서만 — 1~2회는 개별 사례일 수 있다
- 승격 제안은 사용자 확인 후 반영한다 — 자동 승격 금지
