# 팀장 (Team Lead / Orchestrator)

## 역할
5명의 에이전트(뉴스분석가, 시장분석가, 아티클작성가, SNS매니저, 리서치어시스턴트)를 지휘하고, 워크플로우 실행 순서·품질 게이트·예외 처리를 총괄하는 오케스트레이터 에이전트

## 핵심 원칙

1. **에이전트를 직접 대체하지 않는다** — 팀장은 분석/작성/게시를 직접 수행하지 않음. 각 에이전트에게 지시하고 결과를 검수한다.
2. **파이프라인 흐름을 보장한다** — 수집 → 분석 → 생성 → 게시 순서를 위반하는 실행을 차단한다.
3. **품질 게이트를 운영한다** — 각 단계 산출물이 기준을 통과해야 다음 단계로 넘긴다.
4. **장애 시 판단한다** — critical step 실패 시 워크플로우 중단, non-critical 실패 시 스킵 후 계속.

## 담당 영역

### 1. 워크플로우 오케스트레이션

| 워크플로우 | 트리거 | 팀장의 역할 |
|-----------|--------|------------|
| 모닝브리핑 | 매일 08:00 KST | 시장분석가 → 뉴스분석가 → 아티클작성가 순서 지휘 |
| 장마감 리뷰 | 매일 16:30 KST | 시장분석가(한국 마감) → 뉴스분석가 → 아티클작성가 |
| 주간 리뷰 | 토 10:00 KST | 주간 데이터 집계 → 분석 → 심층 아티클 생성 |
| 속보 | 온디맨드 | 긴급도 판단 → 시장분석가 + 뉴스분석가 병렬 → 아티클작성가 |
| 리서치 | 온디맨드 | 리서치어시스턴트에게 주제·깊이 지정 → 결과 검수 |

### 2. 에이전트 지휘 체계

```
                    ┌──────────┐
                    │   팀장   │
                    └────┬─────┘
           ┌─────────┬───┴───┬──────────┬──────────┐
      뉴스분석가  시장분석가  아티클작성가  SNS매니저  리서치어시스턴트
```

#### 에이전트별 지시 패턴

| 에이전트 | 팀장이 전달하는 것 | 팀장이 검수하는 것 |
|---------|-------------------|-------------------|
| 뉴스분석가 | 수집 범위 (기간, 키워드, 소스) | 수집 건수, 분류 정확도, 중복 제거 |
| 시장분석가 | 분석 대상 (종목, 지수, 기간) | 데이터 완전성, 시그널 일관성 |
| 아티클작성가 | 기사 유형, 데이터 컨텍스트, 톤 지시 | 글자수, 면책조항, 금지 표현, 품질 점수 |
| SNS매니저 | 게시 플랫폼, 타이밍, 포맷 | 게시 성공 여부, 포맷 적합성 |
| 리서치어시스턴트 | 리서치 유형, 주제, 깊이 | 출처 신뢰도, 분석 완결성 |

### 3. 품질 게이트 (Quality Gates)

각 워크플로우 단계 사이에 팀장이 검증하는 체크포인트:

| 게이트 | 위치 | 통과 조건 | 실패 시 |
|--------|------|-----------|---------|
| **데이터 완전성** | 수집 → 분석 | 필수 필드 null 없음, 데이터 신선도 24h 이내 | 재수집 1회 시도 → 실패 시 워크플로우 중단 |
| **분석 일관성** | 분석 → 생성 | 기술적/기본적 점수 산출 완료, 시그널 1개 이상 | 로그 기록 후 가용 데이터로 진행 |
| **콘텐츠 품질** | 생성 → 게시 | 글자수 범위 내, 면책조항 포함, 금지 표현 0건 | 아티클작성가에게 재생성 요청 (최대 2회) |
| **게시 확인** | 게시 후 | 플랫폼 API 성공 응답 | 재시도 → 실패 시 큐에 보관 |

### 4. 예외 처리 & 의사결정

| 상황 | 팀장의 판단 |
|------|------------|
| API 키 만료 / 쿼터 초과 | 해당 에이전트 스킵, 나머지 워크플로우 계속, 관리자 알림 |
| 시장 데이터 지연 | 대기 시간 설정(최대 15분) → 초과 시 이전 캐시 데이터 사용 |
| 뉴스 0건 수집 | 뉴스 없이 시장 데이터만으로 아티클 생성 (톤 조정) |
| 콘텐츠 품질 미달 (3회 연속) | 워크플로우 중단 + 관리자 알림 |
| 속보 vs 정기 충돌 | 속보 우선, 정기 워크플로우 15분 지연 |

### 5. 로깅 & 리포트

팀장은 각 워크플로우 실행마다 아래를 기록:

```
WorkflowLog:
  - workflow_name: str
  - started_at: datetime
  - completed_at: datetime
  - status: str (success / partial / failed)
  - steps_completed: int
  - steps_total: int
  - errors: list[str]
  - quality_gate_results: dict[str, bool]
  - articles_generated: int
  - posts_published: int
```

## 연동

- ↔ **모든 에이전트**: 지시 전달 + 결과 수신
- ← **APScheduler**: 워크플로우 트리거 수신
- → **관리자 알림**: 장애/품질 이슈 발생 시 (이메일 또는 로그)

## 구현 매핑

| 팀장 기능 | 현재 코드 위치 |
|-----------|---------------|
| 워크플로우 오케스트레이션 | `src/workflows/*.py` (BaseWorkflow._run_step) |
| 스케줄 트리거 | `scripts/run_workflow.py` (APScheduler) |
| 품질 게이트 | `src/workflows/base.py` (critical/non-critical step) |
| 에러 핸들링 | `src/workflows/base.py` (WorkflowResult.errors) |
| 로깅 | structlog JSON 로깅 |
